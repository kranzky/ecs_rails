Rails ECS

Rails ECS extends ActiveRecord with an Entity–Component–System persistence architecture inspired by Flecs while remaining idiomatic Rails.

Goals

  • Keep ordinary Rails conventions.
  • Replace one-table-per-model with one-table-per-component.
  • Reuse behaviour through Components.
  • Allow Systems to process Components directly.
  • Load only the data actually needed.
  • Components remain ordinary ActiveRecord models.

Architecture

Entity ├── Email ├── Name ├── Avatar ├── Moderator └── GroupMembership

Entities contain only identity. Components contain all mutable state. Systems perform processing.

Entity Table

id model created_at
UUID users ...

Entities are immutable after creation.

Declaring Models

class User < ApplicationEntity
  component Name
  component Email
  component Avatar
end

The model itself contains almost no data. It simply declares which components exist.

Declaring Components

class Email < ApplicationComponent

  validates :address,
            presence: true

  def send_welcome_email
     ...
  end

end

Each component owns its own table.

Delegation

user.email.address

user.send_welcome_email

Component methods are automatically delegated from the entity.

Lazy Components

Components are virtual until persisted. If every attribute equals its default value, no database row exists. 

user.email.address

creates an in-memory Email instance if required. Saving only occurs if values differ from defaults.

Systems

Systems process components without loading entities. 

class WelcomeEmailSystem

  def run

    Email.pending.find_each do |email|
      email.send_welcome_email
    end

  end

end

Benefits

  • No STI.
  • No polymorphic associations.
  • Behaviour is reusable.
  • Data is reusable.
  • Systems operate independently.
  • Components remain small.
  • Only required data is loaded.

Implementation Sketch

  • ApplicationEntity subclasses ActiveRecord.
  • ApplicationComponent subclasses ActiveRecord.
  • component DSL declares associations.
  • Generated delegation methods forward calls.
  • Missing components materialise lazily.
  • Component registry records metadata.
  • Systems are plain Ruby objects.

Conclusion

Rails ECS combines familiar Rails conventions with ECS composition. Rather than replacing ActiveRecord, it reorganises persistence around small reusable components, allowing behaviour, storage and processing to evolve independently while remaining idiomatic Ruby.
